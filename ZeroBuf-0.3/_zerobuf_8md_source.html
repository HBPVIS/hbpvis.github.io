1 # Zerobuf Binary Encoding {#Binary}
2 
3 Nomenclature: A Zerobuf object is the instantiation of a class compiled
4 from a schema file by the Zerobuf compiler. A zerobuf (lower case) is
5 the memory buffer of a Zerobuf object. It is a single memory allocation,
6 i.e., a pointer with a size, and can be serialized as is without any
7 additional transformation.
8 
9 Zerobuf objects save all their member data in a single memory buffer in
10 host endianness. It is the receiving node's responsibility to correct
11 the endianness in a received zerobuf. The zerobuf binary format doubles
12 as the serialization protocol and needs to be implemented by all
13 language bindings correctly. Implementations may differ in their memory
14 allocation strategy.
15 
16 The zerobuf binary format is as follows, and will be explained below:
17 
18  [version][dynamic storage headers][static storage][dynamic storage]
19  dynamic storage headers: 8b offset, 8b size
20  for all dynamic objects in order of spec
21  implemented: dynamic arrays of builtins and static objects, strings
22  not implemented: arrays of dynamic objects, dynamic sub-classes
23  static storage: 1,2,4,8,16b (* static array size)
24  for all static arrays and variables in order of spec
25  Builtin types are: int (4b), uint (4b), float (4b), double (8b),
26  byte (1b), short (2b), ubyte (1b), ushort (2b), ulong (8b),
27  uint8_t (1b), uint16_t (2b), uint32_t (4b), uint64_t (8b),
28  uint128_t (16b), int8_t (1b), int16_t (2b), int32_t (4b), int64_t
29  (8b), bool (1b)
30  dynamic storage layout is an implementation detail of the Allocator
31 
32 A zerobuf always contains a static storage area, and optionally a
33 dynamic storage area for objects with dynamically-sized members. The
34 static storage section is at the beginning of the zerobuf. It contains a
35 version, followed by the headers for dynamic members and the storage for
36 static data. The number of dynamic storage headers and the layout of the
37 static storage section follows directly from the schema. The schema is
38 defined by the .fbs input file and available as the Schema structure
39 from each Zerobuf object.
40 
41 The version is used to detect zerobuf format changes and the producer
42 endianness.
43 
44 The dynamic storage headers contain an offset and size to point to
45 memory allocation in the dynamic storage section. The offset is in bytes
46 from the beginning of the zerobuf. The size is in bytes. The Allocator
47 may create holes during allocations in the dynamic storage section. A
48 dynamic allocation is either a dynamic array or a (recursive) zerobuf
49 managed by a sub allocator. Array allocations only store their data in
50 the allocations. A dynamic Zerobuf sub-object stores a full zerobuf, as
51 specified here. Strings ([string] in .fbs) are handled as a dynamic char
52 array.
53 
54 The static storage section contains all members whose allocation size
55 can be determined at schema compile time. This includes builtin types
56 and static-sized Zerobuf objects, as well as static-sized arrays of the
57 aforementioned. 128-bit integers are exposed through servus::uint128_t
58 in the generated C++ API. All other types are exposed by their
59 corresponding C++ type.

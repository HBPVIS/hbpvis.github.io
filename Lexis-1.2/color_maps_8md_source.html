1 Color maps {#colorMaps}
2 ============
3 
4 Being able to specify a mapping between scalar variables and parameters used
5 for represent visually those variable is a recurring theme in visualization
6 applications. An exemplary case is the transfer functions that map density to
7 RGBA and opacity values in volume rendering with an emission-absorption optical
8 model.
9 
10 This specification describes the messages to deal with maps as a particular
11 case of transfer function (TF). The more generic case will be addressed in a
12 separate document.
13 
14 This document follows the discussion initiated in by [https://github.com/HBPVIS/Lexis/pull/19]
15 
16 ## Requirements
17 
18 - Represent RGBA color maps with an independent TF for each channel. The basic
19  TF definition should be reusable for the more generic case.
20 - Represent extra material properties such as light emission and contribution.
21 - The network representation shouldn't restrict the options in the
22  implementation of a GUI editor.
23 
24 ## Discussion
25 
26 Defining a color map with control points only makes sense if the application
27 shader is capable of computing its values at runtime. In other words, the shader
28 has to be able to compute any value that resolves the linear/Gaussian/etc
29 function defined by control points. That (heavy and incompatible with real-time
30 rendering) computation has to be implemented for all technologies used by the
31 shaders (currently c++, glsl, cuda, ispc, etc). If generalization is wanted in
32 the form of a method of the color map event (implemented in C++), that method
33 has to compute the transformation from the control points to a lookup table in
34 order to feed the shaders. And in that case, why not simply expose the color map
35 as a simple lookup table, and leave the control point management to the client
36 application?
37 
38 ### Why lookup tables?
39 
40 #### Pros
41 - Generated by the UI.
42 - Used as is by the rendering engines, avoiding any kind of transformation in
43 various technologies. Precision is defined by the sampling.
44 - More suitable for real-time rendering since value are precomputed
45 
46 #### Cons
47 - Not as accurate as control point since the precision depends on the sampling
48 rate
49 
50 ### Why control points?
51 
52 #### Pros
53 - Precision is infinite if the application can handle them, otherwise a single
54 C++ implementation in the event does the job but looses the interest of using
55 control points.
56 - Convenient for the UI.
57 
58 #### Cons
59 - A lookup table is still required for convenience with existing (python or
60 JavaScript) client libraries.
61 - Sending control points is not enough to define the transfer function. Other
62 information such as tangents and type of algorithm (guassian, linear, etc) also
63 need to be communicated to the renderer.
64 
65 ### State owner
66 
67 The UI holds the state of the transfer function, and overwrites any value held
68 by the renderer.
69 
70 ### To summarize
71 
72 - The transfer function event is a set a lookup table (RGBA, and extra
73 attributes) with variable lengths and a range of values to which the lookup
74 table is applied.
75 - The size of the lookup table determines the precision of the sampling.
76 - The UI (or any client in general) is responsible for the state of the color
77 map.
78 - The UI is responsible for generating the lookup tables, as well as the
79 sampling rate according to its own capabilities (Control points, mathematical
80 functions, custom color maps, etc).
81 
82 ## Design
83 
84 A simple event defines a range and a transfer function for each 'channel'. In
85 our case, channels are the diffuse color, the opacity (alpha), the light
86 emission, and the contribution to the geometry (how much of the geometry
87 material is impacted by the transfer function).
88 Each lookup table can have a variable number of values.

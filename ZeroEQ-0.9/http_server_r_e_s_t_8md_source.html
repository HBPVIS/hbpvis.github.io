1 HTTP Server for generic REST interfaces {#httpserverrest}
2 ============
3 
4 This document describes a proposed extension to the current http::Server to make
5 it usable as a general-purpose REST interface.
6 
7 ## Requirements
8 
9 * Support all standard http verbs: GET, POST, PUT, PATCH, DELETE
10 * Support custom response payload for all action verbs (not just for GET)
11 * Allow specifying the MIME type of the response to serve any type of content
12 * Allow specifying extra HTTP header fields in response to certain requests
13 * Allow asynchronous processing of requests (not blocking the main thread)
14 * Single endpoint for collections (e.g. windows/, files/, neurons/, etc. )
15 * Allow filtering operations on collections (windows/selected, files?limit=10)
16 
17 ## Dependency Changes
18 
19 None, cppnetlib 0.13 already offers the required functionality.
20 
21 # Proposition #1
22 
23 ## API
24 
25 To fullfill the requirements with minimal API changes, we propose a generic
26 interface as follows:
27 
28  namespace zeroeq
29  {
30  namespace http
31  {
32  enum class Verb { GET, POST, PUT, PATCH, DELETE };
33 
34  // copied from HTTPServer::connection::status_t
35  enum Code { OK = 200, NOT_FOUND = 404, ... };
36 
37  // some commonly used headers, see:
38  // https://en.wikipedia.org/wiki/List_of_HTTP_header_fields
39  enum class Header{ LOCATION, RETRY_AFTER, ... };
40 
41  struct Response
42  {
43  Code code = Code::OK;
44  std::string contentType;
45  std::string payload;
46  std::map<Header, std::string> headers;
47  };
48 
49  /** HTTP REST callback with payload, returning a Response future. */
50  using RESTFunc = std::function<std::future<Response>(const std::string&)>;
51 
52  /** HTTP REST callback with path + payload, returning a Response future. */
53  using RESTPathFunc = std::function<std::future<Response>(const std::string&,
54  const std::string&)>;
55 
56  class Server
57  {
58  public:
59  /** ... */
60 
61  /** Handle a single action on a given endpoint. */
62  void handle(Verb action, const std::string& endpoint, RESTFunc func);
63 
64  /** Handle all urls starting with the given endpoint. */
65  void handlePath(Verb action, const std::string& endpoint,
66  RESTPathFunc func);
67 
68  /** ... */
69  }
70  }
71  }

74 ## Examples
75 
76 The following example shows how the API could be use to expose a collection of
77 windows. For illustration purposes, both handle() and handlePath() are used.
78 Note how the asychronous API is used to process the most time consuming requests
79 without blocking the main thread.
80 
81  using namespace zeroeq;
82  http::Server server{ /** ... */ };
83 
84  void makeThumbnail(const std::string& windowId)
85  {
86  auto response = http::Response{};
87  response.contentType = "image/jpeg";
88  response.payload = doTheHeavyWork(windowId);
89  return response;
90  }
91 
92  // simplified window collection for the example
93  std::set<std::string> windows{"1234", "567"};
94 
95  // Register an individual endpoint for each thumbnail. Note that this is
96  // tedious to manage as the list of windows evolves. It could be more
97  // efficiently served by a generic function:
98  // server.handlePath(http::Verb::GET, "api/windows", getWindowInfo );
99  for (const auto& windowId : windows)
100  {
101  const auto makeThumbnailAsync = [windowId](const std::string&)
102  {
103  return std::async(std::launch::async, makeThumbnail, windowId);
104  };
105  server.handle(http::Verb::GET, "api/windows/"+windowId+"/thumbnail",
106  makeThumbnailAsync);
107  }
108 
109  // Here we use a generic endpoint for all window close operations. "path"
110  // provides the url part after the endpoint: "api/windows/567" -> "567"
111  const auto closeWindow = [&](const std::string& path, const std::string&)
112  {
113  if (!windows.count(path))
114  return make_ready_future(http::Response{http::Code::NOT_FOUND});
115 
116  server.remove("api/windows/"+path+"/thumbnail");
117  windows.erase(path);
118  return make_ready_future(http::Response{http::Code::OK});
119  };
120  server.handlePath(http::Verb::DELETE, "api/windows", closeWindow);
121 
122  // This function uses POST to open a new window and return its location.
123  const auto openWindow = [&](const std::string& request)
124  {
125  if (!windows.insert(request))
126  return make_ready_future(http::Response{http::Code::NO_CONTENT});
127 
128  auto response = http::Response();
129  response.code = http::Code::CREATED;
130  response.headers[http::Header::LOCATION] = "api/windows/"+request;
131  return make_ready_future(std::move(response));
132  };
133  server.handle(http::Verb::POST, "api/windows", openWindow);

135 ## Implementation
136 
137 In requestHandler.h, the struct HTTPRequest needs to be changed from:
138 
139  // output from zeroeq::http::Server
140  HTTPServer::connection::status_t status;
141  std::string reply;
142 
143 to
144 
145  // output from zeroeq::http::Server
146  std::future<Response> response;
147 
148 In requestHandler.cpp, the ConnectionHandler must wait on response.get();
149 
150 In server.cpp, all existing handle*() functions must be adapted to wrap the
151 response data as ready futures. Additional maps must be added for the POST,
152 PATCH and DELETE actions.

154 # Proposition #2
155 
156 This proposition makes use of cppnetlib's internal thread pool to execute the
157 asynchronous functions, removing some complexity from user code.
158 
159 ## API
160 
161 Same enums and Response struct as in proposition #1:
162 
163  namespace zeroeq
164  {
165  namespace http
166  {
167 
168  /** HTTP REST callback with payload, returning an http Response. */
169  using RESTFunc = std::function<Response(const std::string&)>;
170 
171  /** Returned by a RESTPathFunc. Stores the actual RESTFunc to execute
172  and a boolean indicating if it must be run asynchronously. */
173  struct PathResponse
174  {
175  RESTFunc func;
176  bool async = false;
177  };
178 
179  /** HTTP REST callback with path + payload, returning a PathResponse.
180  "path" provides the url part after the endpoint, for instance:
181  endpoint: "api/windows" | url: "api/windows/567/thumbnail" ->
182  path: "567/thumbnail"
183  */
184  using RESTPathFunc = std::function<PathResponse(const std::string&,
185  const std::string&)>;
186 
187  class Server
188  {
189  public:
190  /** ... */
191 
192  /** Handle a single action on a given endpoint. */
193  void handle(Verb action, const std::string& endpoint, RESTFunc func,
194  bool async);
195 
196  /** Handle all urls starting with the given endpoint. */
197  void handlePath(Verb action, const std::string& endpoint,
198  RESTPathFunc func);
199 
200  /** ... */
201  }
202  }
203  }

206 ## Examples
207 
208 This example is similar to the one in proposition #1. It explores the handling
209 of path requests in a synchronous or asynchronous manner depending on the url.
210 
211  using namespace zeroeq;
212  http::Server server{ /** ... */ };
213 
214  // simplified window collection for the example
215  std::set<std::string> windows{"1234", "567"};
216 
217  // example path: "", "567/thumbnail", "567/wrongproperty"
218  const auto getWindowInfo = [&](const std::string& path, const std::string&)
219  {
220  // GET "api/windows" -> list all windows
221  if (path.empty())
222  {
223  const auto func = [&](const std::string&)
224  {
225  auto response = http::Response;
226  response.contentType = "application/json";
227  response.payload = toJsonString(windows);
228  return response;
229  }
230  return PathResponse{func, false};
231  }
232 
233  const auto input = split(path,'/');
234  const auto& windowId = input[0];
235  const auto& property = input.size() == 2 ? input[1] : std::string();
236 
237  if (!windows.count(windowId))
238  {
239  return PathResponse{[](const std::string&)
240  { return http::Response{http::Code::NOT_FOUND}; }, false};
241  }
242  if (property == "thumbnail")
243  {
244  const auto makeThumbnail = [windowId](const std::string&)
245  {
246  auto response = http::Response{};
247  response.contentType = "image/jpeg";
248  response.payload = doTheHeavyWork(windowId);
249  return response;
250  };
251  return PathResponse{makeThumbnail, true};
252  }
253  return PathResponse{[](const std::string&)
254  { return http::Response{http::Code::NOT_IMPLEMENTED}; }, false};
255  }
256  server.handlePath(http::Verb::GET, "api/windows", getWindowInfo);
257 
258  // Here we use a generic endpoint for window close operations. "path"
259  // contains the url part after the endpoint: "api/windows/567" -> "567"
260  const auto closeWindow = [&](const std::string& path)
261  {
262  if (!windows.count(path))
263  return http::Response{http::Code::NOT_FOUND};
264 
265  server.remove("api/windows/"+path+"/thumbnail");
266  windows.erase(path);
267  return http::Response{http::Code::OK};
268  };
269  const auto deleteWindow = [&](const std::string& path, const std::string&)
270  {
271  const auto func = [path](const std::string&){ closeWindow(path); };
272  return PathResponse{func, false};
273  };
274  server.handlePath(http::Verb::DELETE, "api/windows", deleteWindow);
275 
276  // This function uses POST to open a new window and return its location.
277  const auto openWindow = [&](const std::string& request)
278  {
279  if (!windows.insert(request))
280  return http::Response{http::Code::NO_CONTENT};
281 
282  auto response = http::Response();
283  response.code = http::Code::CREATED;
284  response.headers[http::Header::LOCATION] = "api/windows/"+request;
285  return response;
286  };
287  server.handle(http::Verb::POST, "api/windows", openWindow, false);

289 ## Implementation
290 
291  In requestHandler.h, the struct HTTPRequest needs to be changed from:
292 
293  // output from zeroeq::http::Server
294  HTTPServer::connection::status_t status;
295  std::string reply;
296 
297  to
298 
299  // output from zeroeq::http::Server
300  Response response;
301  RESTFunc asyncResponse;
302 
303  In requestHandler.cpp, ConnectionHandler::_handleRequest must do:
304  if (httpRequest.asyncResponse)
305  response = httpRequest.asyncResponse();
306 
307  In server.cpp, additional maps must be added for the POST, PATCH and DELETE
308  actions. Each map must also store a boolean indicating whether the function
309  should be called synchronusly or asynchronously.
310  Adding support for handlePath() should be straighforward.

312 ## Issues

314 ### 1: How many threads should the cppnetlib server use?
315 
316 _Resolution: Open:_ The server thread pool consists of a single thread by
317 default. Using more threads may improve performance by accepting multiple
318 concurrent requests but would require changes in the zeroeq inproc signaling
319 mechanism.

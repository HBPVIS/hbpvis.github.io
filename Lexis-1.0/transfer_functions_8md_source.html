1 Color maps {#colorMaps}
2 ============
3 
4 Being able to specify a mapping between scalar variables and parameters used
5 for represent visually those variable is a recurring theme in visualization
6 applications. An exemplary case is the transfer functions that map density to
7 RGBA and opacity values in volume rendering with an emission-absorption optical
8 model.
9 
10 This specification describes the messages and C++ helper classes to deal with
11 color maps as a particular case of transfer function (TF). The more generic
12 case will be addressed in a separate document.
13 
14 ## Requirements
15 
16 - Represent RGBA colormaps with an independent TF for each channel. The basic
17  TF definition should be reusable for the more generic case.
18 - The C++ helper class will encapsulate a colormap and allow the conversion of
19  the network representation into a OpenGL ready lookup table for creating a
20  texture.
21 - The network representation shouldn't restrict the options in the
22  implementation of a GUI editor.
23 
24 ## Design
25 
26 The consensus has been opting for using 2D piece-wise linear functions specified
27 as (unsorted) lists of control points in R^2. An RGBA colormap is an aggregation
28 of 4 of this control point lists. Generic TFs will reuse the definition of the
29 control point list.
30 
31 Control points are always in the absolute scale, that means that the x coordinate
32 refers to the scale of the source variable and the y coordinate refers to the
33 scale of the target attribute. The range of RGBA channels will be [0, 1] unless
34 an application specifies something different (e.g. in physically based TFs)
35 For example, a simple black to white colormap for neuron membrane voltage
36 could be described by the control points (-80, 0), (-10, 1)
37 
38 A color maps cannot be partially updated, the message always describe the entire
39 color map. In cases where the graphical editor allows remapping a color map to
40 a different input range the editor is responsible of finding the adjusted x
41 positions of the control points. For convenience, the editor may store colormaps
42 using normalized coordinates for the x axis and then recalculate the positions
43 when applied to an application.

190 ## Issues
191 
192 ### 1: Why a color map is not described with a lookup table ?
193 
194 _Resolved: Yes_
195 
196 Because a lookup table is a predefined sampling of the transfer function. On
197 the other hand a piecewise function with control points can be more precise
198 for transfer functions with high frecuencies and more lightweight for simple
199 linear functions. Furthermore, depending on the rendering engine a lookup table
200 may not be needed or may not be sufficient (e.g. pre-integrated volume rendering)
201 The C++ API still provides a function to easily convert the message into a
202 lookup table as this is needed by OpenGL.
203 
204 ### 2: Why the colormap uses independent control point lists for each channel ?
205 
206 _Resolved: Yes_
207 
208 Colormap editors typically have RGBA control points, however, advanced transfer
209 function editors have separate curves for each channel. An event with separate
210 control point lists easily allows the latter, while it's still trivial to use
211 it for the former.
212 
213 ### 3: Why the ColorMap::sampleColors functions take a plain pointer instead of using unique_ptr or returning std::vector
214 
215 _Resolved: Yes_
216 
217 This function is intended to fill texture memory to be used in OpenGL or ISPC so
218 it should not make any assumption about how the memory is handled or allocate
219 it itself, leaving the plain pointer as the only alternative.
220 
221 ### 4: ZeroEQ doesn't allow to subscribe multiple instances of a single serializable type neither in the PUB/SUB nor the HTTP interfaces. How does this specification allow the support of multiple colormaps per applications (e.g. as RTNeuron supports in its Python API)
222 
223 _Resolved: Yes_
224 
225 Deferred to object instance support (or equivalent) in ZeroEQ or application
226 specific support.
227 
228 At the application level, the simplest alternative is to use messages as events
229 that request updates on objects stored on the receiver side and add a new field
230 to the messages to identify the target object. Another option is aggregation
231 on bigger application specific objects.
232 
233 ### 5: Why control points x-values are absolute?
234 
235 _Resolved: Yes_
236 
237 Absolute values require additional semantics to describe what is the intent of a
238 colormap. However, normalized values (in the range [0, 1]) are not sufficient
239 from the application point of view (overall if synchronized views are
240 considered), because the absolute range to which the color map maps is still
241 needed and we want the user to choose this range, not the application to decide
242 it automatically.
243 
244 Since absolute values are needed anyway, we opt for the simplest representation:
245 using absolute values.
246 
247 For storage on disk, generic color maps can be stored always with normalized
248 coordinates, and adjusted after loading, but that is up to the front end
249 interface to decide.
250 
251 ### 6: Should the control point lists be sorted by x-value on the wire-protocol?
252 
253 _Resolved: No_
254 
255 Despite the ControlMap C++ ensures that the points will be always sorted, for
256 robustness it should not be assumed that the received points are in order.
257 However it is not totally clear to me that it's possible to reorder the
258 points upon reception without some undesired side-effect.
